1. Introduction
This report describes the development of a hospital appointment chatbot designed to support booking, checking, updating, and cancelling appointments using natural language. The system also retrieves simple hospital-related information from a small knowledge base and demonstrates how Natural Language Processing (NLP), dialogue management, and conversational design principles can support an administrative task through text-based interaction. The focus of the project is not on advanced artificial intelligence but on transparent methods such as TF-IDF similarity and rule-based logic for intent recognition and information retrieval.
Hospitals often face a high volume of routine administrative queries, and patients frequently rely on phone calls or reception desks for appointment information, which can lead to waiting times and limited access. This project investigates whether a conversational interface can reduce friction in these interactions by allowing users to communicate in everyday language, receive immediate clarification, and perform tasks step by step.
1.1 Project Context
Healthcare organisations are increasingly exploring digital technologies that assist with non-clinical workloads. However, many online systems rely on structured web forms, which may be difficult for users unfamiliar with specific procedures or formatting requirements. A conversational interface may offer a friendlier alternative by allowing users to request services using natural language, while still ensuring that essential information is collected in a correct sequence. Appointment booking is therefore a suitable domain for exploring how conversational interaction can support users and reduce administrative friction.
1.2 Problem Definition
Traditional booking methods are dependent on staff availability and may be stressful for patients who require quick assistance. Users may not know which information is required or how to phrase queries. The problem addressed in this project is how a chatbot can recognise appointment-related intentions, guide users through multi-step procedures, and retrieve stored appointments, while also supporting simple hospital information questions.
1.3 System Objectives
The system aims to:
•	interpret user messages using similarity-based intent detection
•	guide booking through step-wise conversation
•	store appointments persistently using unique IDs
•	retrieve and update appointments based on user input
•	remember the user’s name during a session
•	answer basic hospital questions using a small CSV knowledge base
The project demonstrates fundamental NLP techniques, multi-turn dialogue design, and rule-based memory, rather than sophisticated machine learning.
1.4 Scope of the Prototype
The chatbot supports administrative appointment management only; it does not provide medical advice or access real patient records. All details are fictional and stored in JSON. The prototype is text-based and operates in a single session without authentication. While realistic in structure, it is not intended for real hospital deployment.

2. Chatbot Architecture
This section explains the structure and implementation of the chatbot, including its modules, conversational flow, and technical decisions. The architecture is modular, consisting of the following Python files and data resources:

	
main.py	message loop, routing, and top-level control
intents.py	intent examples, TF-IDF model, similarity thresholding

nlp_utils.py	preprocessing and name extraction

booking.py	appointment booking and updating flows

storage.py	loading and saving JSON data

state.py	global variables for context and booking state

faq.py	FAQ loading and similarity-based answering

hospital_faq.csv	question/answer pairs

appointments.json	stored appointment data





2.1 Functionality
The chatbot performs several types of tasks:
Booking workflow:
•	collects appointment service, date, and time
•	confirms details before saving
•	generates a unique 6-digit ID
Appointment retrieval and modification:
•	checks an appointment by ID
•	cancels an appointment
•	updates date/time of an existing appointment
Identity handling:
•	detects the user’s name from patterns
•	stores it for the session
•	answers “What is my name?”
Hospital information retrieval:
•	answers questions such as visiting hours or department location
•	selects the most similar entry from a CSV list
Conversational and control features:
•	responds to greetings and “how are you”
•	explains capabilities when asked
•	exits on “bye”
•	allows users to cancel a booking process using “cancel” or “stop”
2.2 Natural Language Processing Implementation
NLP utilities are implemented in nlp_utils.py using NLTK for tokenisation, stop-word removal, and lemmatisation. Preprocessing normalises user input, improving similarity matching. TF-IDF vectorisation is applied in both intent detection and FAQ matching. These techniques are computationally simple and interpretable, making them appropriate for a rule-based educational prototype.
Required NLTK assets are downloaded if missing, making the code portable across environments. The preprocessing pipeline ensures that both user input and stored text share the same representation, improving robustness for similarity comparison.
2.3 Intent Detection
Intent detection uses example sentences defined in intents.py. Each example is preprocessed and vectorised using TF-IDF, and cosine similarity is calculated between the current user input and all example sentences. The highest-scoring intent above a threshold is selected. This approach allows variation in phrasing while maintaining predictable behaviour. A dedicated pattern-based name extraction step ensures that “My name is …” always triggers a naming action, even if similarity scores are low.
The book_appointment intent covers sub-tasks such as checking, cancelling, or updating appointments. These are distinguished through rule-based keyword checks in main.py, providing finer control within the booking domain.
2.4 Dialogue State Management
Booking and updating require multiple inputs (service, date, time). A state machine is implemented using global variables in state.py. The variable booking_state tracks which item of information is required next (ASK_SERVICE, ASK_DATE, ASK_TIME, or CONFIRM). While booking is in progress, all messages are routed to handle_booking_flow() until the process is completed or cancelled.
In the confirmation stage, the chatbot summarises the details and explicitly asks the user to confirm or decline. If confirmed, a new appointment is created or an existing one updated; if declined, the state resets. Users can terminate at any time by typing “cancel”, preventing them feeling trapped in an unwanted dialogue branch.
2.5 Identity and Personalisation
Name extraction uses regex patterns to detect expressions such as “My name is …” or “Call me …”. Detected names are stored in state.name_memory and used in subsequent responses, creating personalised feedback such as “Okay Sneha, let’s book an appointment.” If asked, the chatbot retrieves the stored name or explains that it has not been provided yet. Although minimal, this demonstrates conversational memory and improves interaction quality.
2.6 Persistent Appointment Storage
Persistence is handled in storage.py using a JSON file. On startup, existing appointments are loaded; on booking, updating, or cancelling, the JSON file is rewritten. Each appointment contains a unique ID, name, service, date, and time. The function generate_appointment_id() prevents ID collisions by checking against existing values. This persistence supports multi-session realism, although identity memory does not persist across sessions.
2.7 Informal Information Retrieval
A small CSV file provides frequently asked hospital questions and answers. In faq.py, questions are preprocessed, vectorised, and compared with the user query. If the best match exceeds a similarity threshold, the corresponding answer is returned; otherwise, the chatbot politely declines. This behaviour avoids returning implausible or inaccurate responses. Although limited in domain coverage, this component demonstrates how retrieval-based answering can be integrated with transactional dialogue.
2.8 Justification of Design Choices
TF-IDF similarity and rule-based logic were chosen for transparency, simplicity, and alignment with module objectives. The design avoids opaque machine-learning models, favouring methods whose behaviour can be inspected, understood, and debugged. The state-based approach supports controlled transactional interaction, and modular separation clarifies responsibilities. Using JSON and CSV avoids unnecessary infrastructure while demonstrating persistence and retrieval within realistic constraints.

3. Conversational Design
Conversational design concerns how the chatbot communicates and guides the user, rather than how NLP is implemented. For administrative tasks, clarity, guidance, and error handling are essential. The chatbot incorporates several conversational strategies to support accessible interaction.
3.1 Prompt Clarity
Prompts are short, direct, and clearly framed as questions. When additional structure is required (e.g., date format), examples are provided. This reduces cognitive load and helps users supply valid input first time. Prompts also avoid technical language, favouring everyday sentences that users commonly use when speaking with human receptionists.
3.2 Discoverability and Next-Step Suggestions
From the initial greeting, the chatbot explains its capabilities and indicates how appointment IDs work. After booking, it tells the user how to check, cancel, or update appointments. When input is not understood, the chatbot responds with examples of useful phrases. This makes functionality discoverable without reading documentation and helps users recover from uncertainty.
3.3 Error Handling and Recovery
If users provide incorrect formatting or incomplete requests, the chatbot explains what went wrong and how to correct it. During booking, invalid dates trigger remedial guidance. If users attempt to access appointments without providing an ID, the chatbot requests a six-digit identifier and gives examples. The ability to cancel at any time offers control and reduces frustration, particularly for hesitant or inexperienced users.
3.4 Personalisation and Memory
Remembering and reusing the user’s name adds a conversational dimension. Personalisation increases engagement and demonstrates a simple form of contextual memory. This also signals that the chatbot is responding to the individual rather than producing generic messages.
3.5 Confirmation and Safety
Critical actions such as booking or modifying appointments require explicit confirmation. The chatbot summarises the appointment details and waits for the user to confirm or decline. This prevents accidental actions and mirrors practices used by hospital staff, reinforcing user trust and ensuring clarity.
3.6 Context Awareness and Turn-Taking
State variables restrict the chatbot to the current booking step until the process is complete or cancelled. If the user changes topic, the chatbot reminds them that an appointment is in progress and asks whether they want to continue or cancel. This preserves coherence and ensures that required information is collected in order. It also demonstrates an essential principle of task-oriented conversation.
3.7 Inclusivity and Simplicity
The system uses plain English and avoids assumptions about medical knowledge. The text-only interface works in low-bandwidth and screen reader contexts. The absence of complex commands reduces barriers for users unfamiliar with digital systems. These qualities are relevant for healthcare accessibility, where users vary widely in digital literacy.

4. Evaluation
Evaluation focused on usability, performance of intent detection, and persistence across sessions. Although informal and limited in participant numbers, the evaluation provides insight into the chatbot’s strengths and weaknesses.
4.1 Usability Evaluation
Participants were able to complete booking, checking, updating, and cancelling tasks with minimal instruction. Clear prompts and examples helped users understand required inputs. The confirmation step was especially appreciated, as it allowed corrections. Most difficulties involved date formatting but were resolved by corrective prompts. The ability to cancel a booking mid-flow provided reassurance and reduced frustration.
4.2 Performance and Intent Handling
Intent detection worked reliably for phrasing similar to example sentences but sometimes failed on unusual wording, especially for booking. Because TF-IDF depends on lexical similarity, phrases that used unfamiliar vocabulary triggered fallback responses. FAQ retrieval performed well when queries matched the CSV entries but struggled with indirect phrasing. These limitations are expected for symbolic methods and highlight areas for future improvement.
4.3 Persistence and Reliability
Appointment data persisted across sessions, confirming the effectiveness of JSON storage. Updating and cancelling correctly modified stored entries, and ID uniqueness was consistently preserved. After restarting, appointments remained available, providing realistic continuity.
4.4 Overall Findings
The system successfully demonstrated administrative dialogue, identity handling, confirmation strategies, and retrieval-based answering. The main limitations relate to linguistic variability and knowledge base size rather than fundamental design issues.


5. Discussion
5.1 Reflection on System Performance
The combination of rule-based logic and similarity measures provided functional interaction within a well-defined domain. Users were able to complete tasks, modify entries, and recover from errors through conversational cues. The design achieved the project objective of illustrating core NLP and conversational principles.
5.2 Limitations of the Prototype
The chatbot cannot interpret broader language variation beyond provided examples. Its knowledge base is limited to a small CSV file, and date handling does not validate real availability. Identity memory is session-specific. These constraints reflect prototype design and coursework scope rather than technical failure.
5.3 Ethical and Real-World Considerations
Although no real patient information is used, deploying such a system in practice would require secure authentication, encrypted storage, and compliance with healthcare regulations. The chatbot would need to clearly state that it cannot offer medical advice, ensuring users do not misinterpret administrative information as clinical guidance.
5.4 Future Work
Future work could expand intent examples, integrate supervised models, enrich the knowledge base, support multilingual interaction, and incorporate scheduling logic with real hospital systems. Enhancing date and time interpretation, including natural expressions such as “tomorrow” or “next week,” would improve usability. Voice input could also broaden accessibility.
